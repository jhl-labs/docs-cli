# 프로젝트 개요

> 프로젝트의 목적·문제·사용자·가치를 한눈에
>
> _이 문서는 `docs-cli` 표준 스키마 v1을 따릅니다._

## TL;DR

- `docs-cli` 는 어떤 저장소든 **하나의 표준 문서 스키마**로 `docs/` 폴더를 찍어내는 Go 단일 바이너리다.
- 스캐폴딩 → (AI 에이전트) 채우기 → 검증 → 렌더링을 같은 스키마로 일관되게 수행한다.
- 산출물은 **고정된 프론트매터 + 고정된 챕터 구조**라서, 프로젝트가 달라도 형태가 같고 웹 서비스가 매핑 없이 ingest 할 수 있다.
- 차별점: "예쁜 문서 생성기"가 아니라 **정합성을 기계가 판정하는 표준화 도구**다.

## 문제 정의

프로젝트마다 `docs/` 의 구조·깊이·용어가 제각각이면 다음 고통이 생긴다.

- 신규 기여자가 매번 다른 문서 지도를 학습해야 한다.
- 여러 저장소의 문서를 한 웹 서비스로 모으려면 저장소별 매핑 코드가 필요하다.
- "정의만 해두고 뒤에서 빠지는" 문서(빈 챕터, 끊긴 링크)가 리뷰를 빠져나간다.
- AI 에이전트에게 "문서 채워줘"라고 하면 매번 다른 구조의 결과가 나온다.

`docs-cli` 는 **표준 패턴을 코드로 고정**하고, 그 패턴 준수 여부를 `validate` 로 강제해 이 문제를 닫는다.

## 사용자와 이해관계자

| 역할 | 무엇을 하는가 | docs-cli 로 얻는 것 |
| --- | --- | --- |
| 신규 기여자 | 낯선 코드베이스 파악 | 어느 저장소든 같은 문서 지도 |
| AI 에이전트 | 리버스 엔지니어링·문서 작성 | 챕터·프론트매터가 명시된 프롬프트 |
| 플랫폼/웹팀 | 문서를 웹 서비스로 통합 | 정형화된 XML/프론트매터 |
| 메인테이너 | 문서 품질 게이트 운영 | `validate` 로 기계 판정 |
| 의사결정자 | 산출물 표준화 효과 평가 | 일관된 문서셋 + 커버리지 |

## 핵심 가치와 비-목표

**핵심 가치**

- 일관성: 단일 출처(`internal/schema`)에서 모든 산출물이 파생된다.
- 이식성: 프론트매터 + 챕터 구조가 고정이라 웹 서비스가 그대로 소비한다.
- 강제성: 정합성은 사람 눈대중이 아니라 `validate` 가 판정한다.
- 자동화: AI 에이전트를 비대화형으로 호출해 리버스 엔지니어링을 수행한다.

**비-목표 (의도적으로 하지 않음)**

- 임의 구조의 자유 양식 문서 생성 — 표준 패턴이 목적이다.
- 풀 CommonMark/GFM 렌더 — 표준 문서가 쓰는 부분집합만 지원한다.
- 문서 호스팅 자체 — 산출물(HTML/XML)을 만들고 배포는 Pages/웹 서비스에 맡긴다.
